## Проблемы

- Логи терялись из-за отсутствия транзакционности (веб-воркер мог умереть между выполнением бизнес-логики и записью в ClickHouse).
- Ошибки сети приводили к плохому UX.
- Множество мелких вставок негативно влияло на производительность ClickHouse.

## Решение

### Transactional Outbox Pattern

- Логи сохраняются в outbox (модель `LogEntry`) в рамках транзакции вместе с бизнес-операциями, что гарантирует атомарность.

### Пакетная обработка

- Отдельная Celery задача периодически агрегирует события из outbox и отправляет их пакетами в ClickHouse.
- Пакетирование уменьшает нагрузку и повышает стабильность системы.

### Обработка ошибок

- Встроенные retry-механизмы Celery автоматически повторяют попытки отправки при сбоях.
- Все ошибки структурированно логируются через `structlog`.

### Улучшение ClickHouse-клиента

- Класс переименован в `ClickHouseLogClient` для отражения специализации.
- Список столбцов теперь передается в метод `insert` как параметр (с настройками по умолчанию), что делает клиент универсальным.

### Рефакторинг EventLogProcessor

- Метод `process_events` разбит на специализированные подметоды: выборка и блокировка, подготовка данных, отправка в ClickHouse и обновление статусов.
- Использование Pydantic модели (`EventLogRecord`) вместо неструктурированных кортежей улучшило типизацию и читаемость кода.
- Логика обновления статусов записей вынесена в отдельные методы для упрощения поддержки.
